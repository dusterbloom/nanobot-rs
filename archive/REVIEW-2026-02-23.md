# Nanobot Strategic Review: Making It the Best Local Model Harness

## Context

Nanobot is a Rust-based local-first AI assistant framework in Phase 0 (Foundation) of a 4-phase vision. The developer has shipped 27 commits in 7 days with 1,568+ passing tests, building trio mode (router/specialist/main), content pipeline optimization, instrumentation, and provider routing. The current uncommitted work adds specialist memory (per-domain ring buffer) and pipeline routing.

The system works. But trace data from real sessions reveals friction: tools fail on tilde paths, search is broken (Brave 422), tool guards silently block multi-file reads, 226 session files are invisible to recall, and users retry failed operations 3-4x with no guidance. The architecture is impressive but growing in complexity faster than in reliability.

This review identifies the highest-impact changes to make nanobot the definitive local model harness.

---

## 1. ARCHITECTURAL RECOMMENDATIONS

### 1.1 Split SwappableCore Into Domain Facades

**Why**: 38 fields spanning providers, context, behavior, and identity. Every `/local` swap rebuilds all 38 even when changing one model. Every consumer gets access to everything.

**What**: Decompose into 4 `Arc`-wrapped domain structs:
```
SwappableCore {
    routing: Arc<RoutingConfig>,      // providers, models, is_local
    context: Arc<ContextConfig>,      // token_budget, compactor, hygiene
    behavior: Arc<BehaviorConfig>,    // temperature, anti_drift, no_think
    identity: Arc<IdentityConfig>,    // workspace, sessions, learning
}
```
Hot-swap rebuilds only the changed domain. Consumer functions take `&RoutingConfig` not `&SwappableCore`.

**Files**: `src/agent/agent_core.rs` (definition), `agent_loop.rs` (primary consumer), `router.rs`, `tool_engine.rs`

### 1.2 Extract Iteration State Machine From agent_loop.rs

**Why**: agent_loop.rs is ~4,455 lines. The 5-phase state machine (Preparing/PreCall/Calling/Processing/Executing) is the most critical code but is untestable in isolation because it shares a struct with bus channels and cron services.

**What**: Create `src/agent/iteration.rs` with:
- `IterationPhase`, `IterationOutcome`, `StepResult` enums
- All five `step_*` methods as free functions taking `(&TurnContext, &AgentLoopShared)`
- `run_iteration` standalone driver

Gateway runner, `AgentLoop::run()`, and `process_message()` stay in agent_loop.rs.

**Files**: `src/agent/agent_loop.rs` lines 184-1815 -> new `src/agent/iteration.rs`

### 1.3 Replace Thread Repair With Protocol-Correct Construction

**Why**: `thread_repair.rs` (1,304 lines) runs 6 repair passes on every LLM call because messages are mutated in-place at 20+ sites. Fixing symptoms instead of causes means bugs are invisible until they hit production as 400 errors.

**What**: Introduce `MessageBuilder` enforcing invariants at construction time:
- `push_user(content)` — merges with previous user if consecutive
- `push_assistant(content, tool_calls)` — validates unique IDs
- `push_tool_result(call_id, content)` — validates matching assistant exists
- `inject_system_note(content)` — inserts as user message in correct position

Repair pipeline becomes a debug-only assertion, not a runtime dependency.

**Files**: `src/agent/thread_repair.rs` (current), all `ctx.messages.push(json!(...))` sites in `agent_loop.rs` and `tool_engine.rs`

### 1.4 Unify Error Handling With Recovery Actions

**Why**: Three incompatible error strategies: `ProviderError` enums (errors.rs), `String` returns from Tool::execute(), and `IterationOutcome::Error(String)`. Users see "I encountered an error" with no recovery action. Traces show 3-4 retries per failure.

**What**:
- Change `Tool::execute()` to return `Result<String, ToolError>` with `recovery_hint: Option<String>`
- Surface hints to user in response and to LLM for self-correction
- Wire `ProviderError::is_retryable()` into the iteration machine for automatic retry

**Files**: `src/agent/tools/base.rs` (trait), `src/errors.rs`, `src/agent/tool_engine.rs`

### 1.5 Consolidate Config Explosion

**Why**: 90+ config fields across 15 structs. `ToolDelegationConfig` has 19 fields, many overlapping with `TrioConfig` (18 fields). Individual `strict_*` booleans duplicate what `DelegationMode` already expresses.

**What**:
- Remove individual `strict_*` booleans; `DelegationMode` expresses these
- Merge `TrioConfig` into `ToolDelegationConfig` (trio IS a delegation mode)
- Split config into user-facing (~20 fields: keys, models, channels) and internal tuning (separate `tuning.json`)
- Add load-time validation rejecting contradictions

**Files**: `src/config/schema.rs`, `src/config/loader.rs`

## 2. STRATEGIC RECOMMENDATIONS

### 2.1 Stabilize Trio Router Before Building On It

**Why**: The router has 4 parse fallback strategies (tool-call, strict JSON, lenient CSV, directive pack) because the 8B model's output is unpredictable. Every future system (MAKER, pipeline, specialist memory) inherits this unreliability. 4 fallbacks = 4 places for silent semantic corruption.

**What**:
1. Benchmark with fixed 50-prompt test suite (extend `trio_bench.rs`)
2. Evaluate constrained decoding (GBNF grammar) to eliminate all fallback parsers
3. If staying with LM Studio, use function-calling path exclusively and drop JSON-prompt fallback
4. Define reliability target: >95% accuracy before building pipeline/MAKER on top

### 2.2 Fix Real-World Failures Before Adding Features

**Why**: Daily experience has 5 known failure modes. 1,462 unit tests cover parsing, not the failures users actually hit. The specialist memory and pipeline routing being built now add complexity while basics remain broken.

**Priority order**:
1. **Tool guard default** — `max_same_tool_call_per_turn: 1` blocks multi-file reads. Make per-tool: 5 for read tools, 1 for write tools
2. **Path expansion** — verify router's tool dispatch path calls `expand_path`
3. **Search failure visibility** — structured error when Brave returns 422
4. **Session search** — index JSONL at write time (SQLite FTS5)
5. **Emergency trim visibility** — notify user when messages are dropped

### 2.3 Defer Phase 2-3 Infrastructure

**Why**: `lora_bridge.rs` (737 lines), `step_voter.rs`, `confidence_gate.rs`, `knowledge_store.rs` (681 lines) are dead code. They took real time, will need rewriting when the foundation changes, and create the illusion that features are closer to done than they are.

**What**: Feature-gate with `#[cfg(feature = "experimental")]`. Define Phase 0 completion criteria:
- Trio routing >95% accuracy on benchmark
- Zero known user-facing tool failures
- Session recall works across all sessions
- `/local` hot-swap succeeds 100% without manual intervention

### 2.4 Invest in Integration Tests

**Why**: 1,462 unit tests but zero integration tests. Multi-component failure chains (router misparse -> dispatch error -> thread repair -> dropped question) are invisible.

**What**: Create test harness running complete `process_message` with mock LLM provider. Test:
- Trio end-to-end (router -> specialist -> main synthesis)
- Context overflow -> compaction -> session persistence
- Tool failure -> recovery hint to user

### 2.5 Reduce Config Surface Area

**Why**: 90+ config fields, 30+ default functions. Most are development toggles, not user choices. Users need: API keys, model names, endpoints, delegation mode. The rest should be hardcoded defaults overridable only by advanced tuning file.

## 3. UX RECOMMENDATIONS

### 3.1 Show Trio Decision Chain Live

**Why**: Trio mode has ~3s of silence (router 900ms + specialist 800ms + main 1200ms). User has no idea what's happening. Trace infrastructure writes to JSONL but not to the user.

**What**: Emit status through `text_delta_tx` in dim text:
```
[router] analyzing... (900ms)
[router] -> specialist:summarize (0.92)
[specialist] processing... (800ms)
[main] synthesizing...
```

**Files**: `src/agent/router.rs` (preflight/dispatch), `agent_loop.rs` (text_delta_tx)

### 3.2 Make Error Messages Actionable

**What**: Top 8 error-to-action mappings:

| Error | User message |
|-------|-------------|
| Connection refused | "Local server not responding. Start LM Studio or /restart" |
| Model not loaded | "Model loading, wait 10-30s and retry" |
| Context overflow | "Too long for this model. Use /compact" |
| Tool timeout | "Timed out after Ns. Try simpler command" |
| Permission denied | "Cannot access [path]. Check permissions" |
| Empty response | "No output. Try /think off or rephrase" |
| Search 422 | "Web search unavailable (API issue)" |
| Rate limited | "Rate limited. Waiting [N]s..." |

**Files**: `src/errors.rs` (new `user_facing_error()`), `agent_loop.rs` (IterationOutcome::Error path)

### 3.3 Show Context Pressure in REPL Prompt

**Why**: 32K local context fills without warning. Emergency trim silently drops messages. `/status` exists but requires manual invocation.

**What**: Persistent prompt indicator:
```
[ctx: 18K/32K | trio | gemma-3n] >
```
Yellow at 80%, red at 90%. Print warning when emergency trim fires.

**Files**: `src/repl/mod.rs`, `src/agent/agent_core.rs` (RuntimeCounters)

### 3.4 Make /local Hot-Swap Atomic

**Why**: `/local` rebuilds core, restarts health watchdog, rebuilds agent loop. Partial failure leaves inconsistent state. `rebuild_agent_loop` doesn't report errors.

**What**: Before swap: verify server reachable + model loaded. After swap: single-token probe. On failure: keep current mode with specific error. On success: summary line.

**Files**: `src/repl/commands.rs` (/local handler), `src/server.rs`

### 3.5 Surface Tool Guard Rejections

**Why**: Guard silently blocks 2nd+ `read_file` per turn. User asks "read these 3 files", gets incomplete info, no explanation.

**What**: Inject visible note: `[guard] blocked duplicate read_file (limit: 1/turn)`. Raise default to 3 for read-only tools.

**Files**: `src/agent/tool_guard.rs`, `src/config/schema.rs`

## 5. ANTI-PATTERNS TO STOP

### 5.1 Stop Adding Router Parsing Fallbacks
4 fallback parsers mask model quality problems. The `normalize_action` function silently remaps unknown actions ("summarize" -> "specialist", "clarify" -> "ask_user"). Use constrained decoding or better models instead.

### 5.2 Stop Adding Config Fields Without Removing Others
Every feature gets a toggle. 90+ fields = untestable combinatorial explosion. Make a decision: if a feature should be on, remove the toggle. Use compile-time feature flags for experiments, not runtime config.

### 5.3 Stop Building Phase 2-3 Code During Phase 0
`lora_bridge.rs`, `step_voter.rs`, `confidence_gate.rs`, `knowledge_store.rs` are dead code that will need rewriting. Write designs in ARCHITECTURE.md, not code in src/.

### 5.4 Stop Using `#![allow(dead_code)]` at Module Level
Both `agent_loop.rs` and `tool_runner.rs` suppress the compiler's most useful signal. Apply `#[allow(dead_code)]` per-item only for intentionally-unused code.

### 5.5 Stop Mixing Concerns in TurnContext
28 fields mixing config, channels, conversation state, and tracking. Every step function takes `&mut TurnContext` even when reading 3 fields. Decompose into sub-structs so dependencies are explicit.

9. Trio decision chain visibility (3.1)
